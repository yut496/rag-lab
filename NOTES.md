# RAG最小実装 Q&A まとめ

`rag.py`（最小RAGパイプライン）を読みながら出た疑問と回答の学習ログ。

## Q1. Voyage API は何をしている？

テキストを**意味の座標（ベクトル）に変換**するembedding API。Anthropicはembedding APIを提供しないため、埋め込みだけVoyageを使う。`rag.py` では6チャンクを1リクエストで送り、各チャンクにつき1本のベクトルを受け取る。

## Q2. `zip` は配列を同じインデックスごとにまとめる、で合ってる？

合っている。複数のイテラブルの**同じインデックスの要素をタプルにして**1つずつ返す。

- 注意点: **長さが違うと短い方で止まる**（余りは黙って捨てる、エラーにならない）。検知したいなら `zip(a, b, strict=True)`（Python 3.10+）。
- 使用箇所: `cosine()` の内積計算（`zip(a, b)`）、`build_index()` のチャンクとベクトルの対応付け（`zip(chunks, vectors)`）。

## Q3. Voyage はチャンクごとにベクトル化した数値を返している？

はい。**1チャンク → 1本のベクトル（1024個のfloat）**。

- 個々の数字に意味はなく、**1024個まとめて1つの座標**として意味を持つ。
- 重要な性質: **意味が近いテキストほど座標も近くなる**ように学習されている。
- **文字数が違っても必ず同じ1024次元**に落ちる → だから後で一律に距離比較できる。

## Q4.（最重要）embeddingは座標を計算しているだけで、既存のベクトル空間に埋め込んではいない？

**逆。「既存の固定空間に埋め込む」が正解。**「座標を計算する」と「既存空間に埋め込む」は同じこと。

- `voyage-4-lite` は学習済みで、その重みの中に**1024次元の"意味の地図"が固定的に存在**。APIはテキストが**その地図のどこに来るか**を計算しているだけ。
- 例え: **ジオコーディング**（住所 → 緯度経度）。緯度経度の座標系は既に存在し、住所をその上に点として置く。だから世界中の住所を同じ座標系で比較できる。
- **なぜ重要か**: 全チャンクも質問も**同じ1つの空間**に埋め込まれるからこそ、⑤のコサイン類似度が意味を持つ。

実務の鉄則:

1. **チャンクと質問は必ず同じモデルで埋め込む**。
2. **モデルが違えば空間も違う**（Voyage と OpenAI のベクトルは互換性なし → モデルを変えたら索引を作り直す）。

## Q5. index は、チャンクと出典とベクトル座標を関連付ける作業？

はい。別々だった `chunks`（text, source）と `vectors`（座標）を、`zip` で**同じインデックスどうし束ねて** `{text, source, vector}` の1レコードにする。

- 目的: 検索（⑤）で「ベクトルが近い」→「その本文と出典は？」を即座に辿れるようにするため。
- 「index」の二義: **データ構造の索引**（`build_index` の戻り値＝検索対象の集合）と、**配列の添字**（`chunks[3]` の `3`、`zip` が揃えているのはこっち）。
- 索引の本質は**「(text, source, vector) のただのリスト」**。実運用のFAISS/HNSWも同じことをしていて、違いは**総当たりを高速化する仕組みが乗っているか**だけ。

---

**核**: embedding = 固定された意味空間への配置 / retrieval = その空間内での近傍検索。
次の一手: `CHUNK_SIZE` を変えて再実行し、②chunkingが⑤retrievalのスコアをどう揺らすか（＝「チャンク分割が雑だと正解が割れる」）を手で体感する。
