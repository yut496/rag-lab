# 疑問と回答

## Voyage API は何をしている？

テキストを**意味の座標（ベクトル）に変換**するembedding API。\
`rag.py` では6チャンクを1リクエストで送り、各チャンクにつき1本のベクトル（1024個のfloat）を受け取る。

- 個々の数字に意味はなく、**1024個まとめて1つの座標**として意味を持つ。
- 重要な性質: **意味が近いテキストほど座標も近くなる**ように学習されている。
- **文字数が違っても必ず同じ1024次元**に落ちる → だから後で一律に距離比較できる。

## `zip` は配列を同じインデックスごとにまとめる？

YES、複数のイテラブルの**同じインデックスの要素をタプルにして**1つずつ返す。

## コサイン類似度計算とは？

２つのベクトルがどれだけ同じ方向を向いているかを測る指標。

- 同じ方向 → 角度0° → 類似度 1（最も似ている）
- 直交（無関係）→ 角度90° → 類似度 0
- 真逆 → 角度180° → 類似度 -1

計算式
$$
\cos(\vec{a}, \vec{b}) = \frac{\vec{a} \cdot \vec{b}}{|\vec{a}||\vec{b}|}
$$

- 分子：内積（各要素を掛けて合計）
- 分母：それぞれのベクトルの長さ（大きさ）の積

# ゴールのアウトプット

- **RAGの全体フローと各処理を自分の言葉で説明できる**:
　上から順番に処理が進む。
　ingetion：corpusを取得する。
　chunking：corpusを特定の区切りで分割する。
　embeding：chunkを学習済みベクトル座標に埋め込む。
　retrieval：問いとchunkの類似度を計算する。
　context：generationに渡すcontextに類似度の高いchunkを含める。
　generation：回答を生成する。

- **chunking が retrieval を揺らすことを体感する**:
  `CHUNK_SIZE` を 80 など小さくして再実行し、正解が複数チャンクに割れ・文の途中で切れ、retrievalスコアや回答がどう変わるか。

  CHUNK_SIZE=300の時
  　結論：
  　　retrievalスコア全体でchunkの一致不一致がはっきりと認識できる。
  　　離散的な数値となる。
  　観察：上位数件の類似度が0.5前後の数値、それ以外は0.1前後の数値。
  80の時
  　結論：
  　　retrievalスコア上位の数値は高くなり、また0.5前後の数値をとるchunkの数が増える。
  　　全体は連続的な数値となる。
  　　文脈が保持しにくい。
  　　回答は300の時と変わらず正しかった。
  　観察：
  　　上位数件の類似度が0.6前後でより高い数値を示した。
  　　全体として眷属的な数値を示した。
  　　chunkが短い分、文脈が削られる。
- **input_type の非対称埋め込みを理解する**:
  embedingは文書を `document`、retrievalは問いを `query` で埋める。
  同じテキストでも「文書として」と「問いとして」で埋め方が変わる。

  なぜか
  同じ意味の文章でも、質問として組み立てられたものか、文書として組み立てられたものかで表現が大きく異なる。
  そのため、モデルは同じ意味の質問と文章のベクトル座標を近くするようにあらかじめ学習させるようにしている。
  なので、適切なinput_typeを選択しないと正確なベクトル座標が算出できない。

- **retrieval が正解チャンクを上位に引けることを確認する**:
  retrievalのコサイン類似度ランキングで、答えを含むチャンクが上位に来るか。来なければ「検索の失敗」。

　以下の問いを投げた時に、azure_pangolin.txtのchunkに対する類似度が上位に来ることを確認した。
  How many termites can the azure pangolin eat in a night?

- **grounding（捏造抑制）が働くことを確認する**:
  コーパスに無い質問
  （`uv run python minimal-rag/rag.py "What is the capital of France?"`）で、無関係なチャンクが
  引かれ Claude が「分からない」と答える。

　generation時のsystem promptとして、groundingを含めることでcorpusに存在しない質問にはわからないと回答することを確認した。
