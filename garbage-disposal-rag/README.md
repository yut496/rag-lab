## Purpose
現実のコーパスと精度向上機構を含むRAGシステムを、単一自治体スコープ
（大田区・ルール解説のみ・単一テナント）で構築する。その過程で必要と
なる観点・思考と、仮想運用で顕在化する課題を、再現可能な記録として
観察・言語化する。さらに、顕在化した課題に対して改善案を立案・適用し、
同一の評価セットでの再測定によってその有効性を検証する。
プロダクト化（認証/多テナント/監視）はスコープ外。

## Definition of Done（全項目が二値で「済」になったら完了）

### 0. タイムボックス（着手前に確定）
- [ ] 全体予算 = 正味24h
- [ ] フェーズ別想定: 環境3h / コーパス整形10h / GS構築4h / 検索・棄却3h / 評価・改善4h
- [ ] 末尾4h（評価・改善）は構築の遅延に関わらず確保する（構築側を削る）
- [ ] サブ・タイムボックス: コーパス整形が10hを超えたら現品質で凍結
- [ ] 超過時の削減順: grounded採点を落とす → GSを10問に縮小 → ソースpage削減
      （失敗想定の積み込み・改善1サイクルは削らない）

### 1. ゴールデンセット（着手後・改善サイクル前にフリーズ）
- [ ] 計15問を原文のまま凍結。内訳を満たす:
      - 実問い合わせ系 10問（公式FAQ/業者FAQ/自作口語、出どころタグ付き）
      - 合成 5問
      - うち「棄却すべき問い（分別判定/収集日）」を最低3問含む
      - 「コーパスに無い品目」を最低1問含む（例: モバイルバッテリー）
      - 「R7.4改定で正解が変わった内容」を最低1問含む
- [ ] 各問に正解ラベルを付与: expected_category / should_abstain
      （理想回答は任意で添えるが、一致は正解条件にしない）
- [ ] スモーク5問 / 本評価15問 の二層に分ける
- [ ] フリーズ後、改善サイクル中に質問セットを変更しない

### 2. オフライン評価（パイプライン変更ごとに回帰実行）
- [ ] 3軸を分離して手採点: retrieval_ok / grounded_ok / abstain_ok
- [ ] 結果を results.csv に記録（id, routed, top_score, top_category, 3軸）
- [ ] grounded は「回答が参照チャンクに無いことを述べていないか」で判定
      （理想回答との文面一致では判定しない）

### 3. 失敗の言語化
- [ ] 失敗モードを証拠（質問ID＋実際の出力）付きで最低3つ文書化
- [ ] 各失敗を5原因のどれかに仮説マッピング

### 4. 改善サイクル（最低1回）
- [ ] 最悪failを1つ選ぶ
- [ ] 原因仮説を1つ立てる（5原因のどれか）
- [ ] 変更を1つだけ加える（単一変更）
- [ ] 同一フリーズセットで再実行し、3軸の差分を記録
- [ ] 差分から改善案の有効性を判定（有効/無効/悪化/判定不能のいずれか）

### 5. 観察記録
- [ ] フェーズ別の実測工数を記録し、想定との比較表を作る
- [ ] 詰まった判断・捨てた選択肢をログ
- [ ] コーパス取得日（スナップショット日付）を記録
