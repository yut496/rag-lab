# lost in the middle（詳細）

> [GLOSSARY](../GLOSSARY.md) の `lost in the middle` の詳細ページ。

## 定義

**長いコンテキストを与えられた LLM が、入力の「中間」に置かれた情報を見落としやすい現象。**
2023年の論文 *"Lost in the Middle: How Language Models Use Long Contexts"* (Liu et al.) で命名され広く知られるようになった。

複数の文書をコンテキストに詰め込み、そのうち1つに正解が含まれるタスク（質問応答など）を解かせると、**正解文書の位置によって精度が大きく変わる**。

## U字カーブ

精度を「正解文書の位置」に対してプロットすると、きれいな **U字** を描く。

```
精度
 高 ┤●                              ●
    │ \                            /
    │  \                          /
    │   \____                ____/
 低 ┤        ‾‾‾‾●______●‾‾‾‾
    └──────────────────────────────── 正解の位置
      先頭        中間         末尾
```

- **先頭（primacy 効果）**: 高精度
- **末尾（recency 効果）**: 高精度
- **中間**: 大きく精度低下 ← これが "lost in the middle"

衝撃的だったのは、「中間に正解があるくらいなら、その文書をコンテキストに**入れない方がマシ**」というケースすらあった点。情報を追加したのに性能が下がるという、直感に反する結果だった。

## 原因（諸説）

1. **位置エンコーディングの偏り** — 多くのモデルが先頭・末尾のトークンに注意（attention）を割きやすい傾向を持つ。
2. **学習データの偏り** — 重要な情報が文書の冒頭・末尾に来やすい自然言語の性質を学習している。
3. **attention の希薄化** — トークン数が増えるほど、中間の各トークンへ向く注意の総量が薄まる。

## RAG との関係

RAG はこの問題に直撃する。典型的な流れは:

1. retrieval（検索）で関連文書を上位 `TOP_K` 件取得する。
2. それらをすべてプロンプトに連結して `context` として LLM に渡す。

このとき、**検索順位どおりに並べると、4位・5位あたりの文書がちょうど「中間」に落ち込んで無視される**可能性がある。せっかく検索でヒットさせても、生成段階で活かされない。

### 対策

| 対策 | 内容 |
|---|---|
| **reranking + 並べ替え** | 最も重要な文書を**先頭か末尾**に配置し、中間に埋もれさせない。 |
| **`TOP_K` を絞る** | コンテキストを短くして中間そのものを減らす。「より多く詰める」より効くことが多い。 |
| **sandwich 配置** | 重要文書を先頭と末尾の両方に置く。 |
| **チャンク粒度の最適化** | 1チャンクを小さくして関連箇所を凝縮する。 |
| **reranker モデルの導入** | cross-encoder 等で検索結果を並べ替え、上位の質を上げる。 |

## 補足

- ロングコンテキストモデル（数十万〜100万トークン級）でも、**程度の差はあれこの傾向は残る**ことが後続研究で報告されている。コンテキスト窓が広いこと ≠ 中間を均等に使えること。
- **Needle in a Haystack（干し草の中の針）** テストは、この中間見落としを測る代表的なベンチマーク。

## 関連用語

- [`retrieval（検索）`](../GLOSSARY.md) — 上位数件を引く工程。並び順がこの現象に影響する。
- [`TOP_K`](../GLOSSARY.md) — 採用件数。絞ることが対策になる。
- [`context（文脈）`](../GLOSSARY.md) — LLM に渡す連結文字列。並べ方が効く。
- [`reranking / cross-encoder`](../GLOSSARY.md) — 並べ替えによる代表的な対策。
